01 — The Engineering Problem

Why does model choice matter so much?

Choosing the wrong architecture for a physical system isn't just an academic mistake — it costs real compute time, real money, and produces models that hallucinate physics they were never taught.

PINN Loss:   L = Ldata + λ · Lphysics ← physics constraint baked into the training objective
Plain language — what this formula means A PINN trains on two signals simultaneously. L_data: how well predictions match actual sensor measurements. L_physics: how much the predictions violate the known governing equation (e.g. Newton's 2nd law, heat conduction). The parameter λ controls the balance between them. Unlike a standard neural network, the PINN is penalised every time a prediction contradicts the physics — making it impossible to "forget" the underlying laws of the system.
Classical NN / Transformer

Data-Driven Approach

Learns patterns purely from data. Powerful generalisation but blind to physical laws. Requires large labeled datasets. Scales with data volume and compute.

Physics-Informed NN (PINN)

Physics + Data Hybrid

Embeds governing equations (PDEs/ODEs) via Lphysics in the loss function. Extrapolates physically. Works with sparse sensor data. Training is harder due to competing, stiff loss gradients (see Appendix A1).

Plain language — what "O(n²·d)" means for your budget Big-O notation describes how cost grows as inputs get larger. O(n²·d) means: double the sequence length (n) → cost quadruples. For a Transformer processing 8,000 tokens with hidden dimension d=64, the attention matrices alone require approximately 8 GB of GPU memory at FP16 precision, batch size 1. This is why long documents are expensive — and why Flash Attention (Dao et al., 2022) was a major practical breakthrough.

Notation key — PINN complexity O(n_c × L × d²) n_c = collocation points (domain interior samples; typically 10k–100k for 3D PDEs) · L = number of network layers · d = hidden layer width. Each collocation point requires evaluating the PDE residual via two backward passes through the full network (computing second-order derivatives like ∂²u/∂x²). This makes each PINN training step approximately 3–5× more expensive than a standard NN training step of equivalent architecture size (Raissi et al., 2019, §3). Full derivation in Appendix A3.
⚠ Transformer Bottleneck

Self-attention is O(n²·d). At n=10k tokens, memory usage becomes prohibitive on standard GPUs. Flash Attention (Dao et al., 2022, NeurIPS) reduces memory to O(n) via IO-aware tiling — same exact output, no approximation.

⚠ PINN Bottleneck

Auto-differentiation for ∂²u/∂x² requires backpropagation through the backpropagation graph (Hessian-vector products). For 50k collocation points × 10 layers: ~10⁹ FLOPs per training iteration, with poor GPU occupancy due to sequential small operations.

04 — Scaling Intuition

When does more = better?

Scaling laws describe how training loss decreases as data, parameters, or compute increase. PINNs and NNs follow different scaling curves — they cross at a regime boundary — and PINNs plateau earlier in the high-data regime due to training stiffness from competing loss gradients.

Low-Data Regime — PINN Wins
Physics acts as a massive implicit regulariser — the model cannot overfit to noise because the PDE loss anchors every prediction. This is the primary engineering use case for PINNs: expensive physical experiments yield only sparse sensor readings, but the governing equations fill in the spatial gaps.
High-Data Regime — NN Wins
Classical NNs and Transformers scale better with data. PINNs plateau earlier — the physics loss creates gradient conflict (Wang et al., 2021, SISC 43(5)). This is the training stiffness problem: two competing objectives prevent the model from squeezing out the final error that a pure data-driven model achieves freely.
Chinchilla Scaling Law Analogy (Hoffmann et al., 2022)

For LLMs: optimal model size ∝ dataset size — scale both proportionally. For PINNs: optimal collocation point count ∝ PDE solution complexity. The principle is identical: do not blindly scale model parameters without proportionally scaling the training signal. Increasing network width without increasing n_c leads to physics overfitting; increasing n_c without wider networks underfits the boundary conditions.

Empirical basis for the scaling chart above The crossover point and plateau behaviour shown in the chart are conceptually derived from: (1) Wang et al. (2021, SISC) who show empirically that PINN loss stagnates due to gradient imbalance once data density exceeds a threshold; (2) Mishra & Molinaro (2023, IMA J. Numer. Anal.) who prove generalisation bounds showing PINN error decays slower than NN error as n → ∞ when physics residuals compete with data fit. The exact crossover data-point is problem-specific and should be benchmarked per system. Do not use this chart as a precise guide without running your own scaling experiments.

🚨 Data Leakage — What It Is & Why It's Catastrophic
  • Definition: When information from the test/validation set contaminates the training process. The model appears better than it is — it is effectively "cheating" on the exam.
  • Case 1 — Normalisation: Computing μ/σ across the entire dataset before splitting. Test distribution is baked into preprocessing. Fix: fit scalers only on training fold, then transform test with the same scaler.
  • Case 2 — Temporal split: In time-series physical systems (e.g., heat flow), a random train/test split leaks future states into training. Fix: always split by time — first 80% of timesteps for training, last 20% for testing.
  • Case 3 — PINN boundary conditions: Using the same boundary condition measurements in both the training physics loss and the validation metric. The model has already "seen" those boundaries. Fix: maintain a held-out boundary test set used only for evaluation.
  • Consequence: Validation loss looks excellent. Real-world performance collapses. Particularly dangerous in safety-critical systems (aerospace, nuclear, medical devices) where field failures are costly or catastrophic.
✓ Best Practice

Always fit preprocessing (scalers, encoders) only on training data. Use temporal or physics-domain-aware splits. Version your data with DVC or equivalent — data provenance is as important as code versioning.

PINN-Specific Rule

Strictly separate collocation points (interior domain, used in L_physics) from boundary condition sensor measurements (used in L_data). These must never overlap with the validation set. A dedicated boundary holdout set is mandatory for honest evaluation.

💡 Memory Bandwidth is the Real Bottleneck

In both architectures, the practical constraint is often memory bandwidth, not raw FLOPs. Matrix multiplications are memory-bound at small batch sizes — the GPU spends more time waiting for data from DRAM than performing arithmetic. PINN's collocation loop creates many small sequential operations, leading to poor GPU occupancy. Always profile with PyTorch Profiler or Nsight Compute before optimising.

Why PINN training steps are 3–5× more expensive A standard neural network gradient step = one forward pass + one backward pass. A PINN must additionally compute second-order derivatives of the output (e.g. ∂²u/∂x²) to evaluate the PDE residual. This requires running backpropagation through the backpropagation graph itself — sometimes called "second-order auto-differentiation." The full cost derivation, including the exact FLOPs multiplier formula, is in Appendix A3.
⚑ Memory note: For self-attention with n=8k, d=64, FP16, batch=1: activation memory for the attention score matrix ≈ n² × 2 bytes = 128M entries × 2 = ~256 MB per head, ×32 heads ≈ 8 GB. FP32 doubles these figures. Flash Attention (Dao et al., NeurIPS 2022) reduces this to O(n·d·√M) where M = SRAM size, via IO-aware blocked computation — no approximation, identical output.
07 — Case Study: Physical System
🌡️
Chosen Physical System

2D Heat Conduction Problem

A steel plate with a fixed heat source at one end and cooling boundary conditions. Classic PDE-governed physical system used in aerospace, electronics thermal management, and manufacturing process control.

∂u/∂t = α (∂²u/∂x² + ∂²u/∂y²) ← 2D heat equation (parabolic PDE)

Boundary conditions: u(0,y,t) = T_hot, u(L,y,t) = T_cold
Initial condition: u(x,y,0) = T_ambient

PINN loss: L = ‖∂û/∂t − α∇²û‖² + λ‖ûBC − uBC,data‖²
Plain language — the heat equation for non-specialists The equation says: how fast temperature changes at any point = a material constant (α, thermal diffusivity of steel ≈ 1.17×10⁻⁵ m²/s) × how curved the temperature profile is around that point. Flat temperature profiles change slowly; sharp gradients (near the heat source) change fast. The PINN learns to predict temperature û(x,y,t) at any spatial location and time, and is penalised at every training step for any prediction that violates this relationship — not just at the 20–50 sensor locations, but at 10,000+ interior domain points simultaneously.

Data & Compute Bottlenecks in this System

  • 1
    Sparse Sensor Data: Physical thermocouples are expensive to install and maintain. A typical experimental setup yields 20–50 point measurements on a plate requiring thousands of spatial coordinates. PINNs are ideal here — the heat equation fills spatial gaps between sensors with physically consistent predictions.
  • 2
    Temporal Leakage Risk: Random-splitting time-series sensor data leaks future temperature states into training. Always split by time: first 80% of timesteps for training, last 20% for testing. Never shuffle a physical time series — sequence matters.
  • 3
    PINN Matrix Op Cost: Computing ∂²u/∂x² via auto-diff requires two backward passes per collocation point. For 50,000 points × 10 layers: ~10⁹ FLOPs per training iteration on top of the standard forward/backward pass. For 3D transient problems, n_c typically needs to exceed 100k for convergence.
  • 4
    NN Width vs. Cost: Capturing sharp thermal gradients near boundaries requires large hidden dimensions (d ≥ 512). Matrix ops scale as O(d²) per layer: a 512→512 layer has 262,144 parameters; a 1024→1024 layer has 1,048,576. Width scales the cost quadratically — choose carefully based on required spatial resolution.
  • 5
    Normalisation Leakage: Temperature normalisation parameters (min, max, or μ/σ) must be estimated from training time-steps only. Using global values from the full simulation run leaks steady-state temperature information into the transient learning phase, inflating apparent validation accuracy.
  • 6
    Causal Training: Standard PINN training applies the physics loss across all time steps simultaneously — causally incorrect for a parabolic PDE. Without causal weighting (Wang et al., 2022), the network is penalised for violating heat flow at t=10s before it has learned the state at t=1s. See Appendix A2.

🏗️ Recommendation for the Heat System

Sparse thermocouple data + hard physics constraints (conservation of energy) + safety requirements → Start with PINN. Once trained and validated, distil to a lightweight classical NN surrogate for fast real-time inference. This hybrid approach satisfies physics compliance during training and latency requirements during deployment.

⚠ Limitations of This Analysis — What We Are Not Claiming
The decision matrix is heuristic. Each row captures one dimension in isolation. Real projects involve multiple simultaneous constraints. A system with sparse data AND limited compute requires different trade-offs than either row alone suggests — consider the full column before choosing.
The compute estimates are order-of-magnitude. The "3–5× PINN cost multiplier" is derived from benchmarks on specific PDE types. For 1D ODEs the overhead may be ×1.5; for turbulent 3D Navier–Stokes it may exceed ×20. Always profile on your specific problem before committing to infrastructure.
PINNs are an active research area. Techniques like NTK-based loss weighting (Wang et al., 2021), causal training (Wang et al., 2022), and NVIDIA Modulus have substantially reduced PINN training difficulty since the Raissi et al. (2019) baseline. The "PINN training is hard" assessment applies to vanilla PINNs, not state-of-the-art implementations.
The hybrid recommendation assumes separable workflows. PINN → NN distillation works well when the PINN can be trained offline. For online learning or continually shifting physical conditions (e.g., sensor degradation, varying material properties), the distillation pipeline requires re-evaluation.
09 — Summary & Takeaways

The 3 Things engineering management needs to know

01

Architecture is a Data Decision

Don't pick NN vs PINN based on hype or familiarity. Ask: how much labeled data is available, and do we need physics compliance? Classical NNs need data volume. PINNs need governing equations. The right answer changes entirely with the data regime.

02

Matrix Ops = Cloud Bill

O(n²) attention and O(n_c × L × d²) PINN auto-diff translate directly to compute cost. PINN steps are 3–5× more expensive than equivalent NN steps due to second-order differentiation. Always profile before optimising — memory bandwidth is usually the real constraint.

03

Data Leakage Kills Silently

Models with leakage look excellent in validation and fail in production. Temporal splits, correct normalisation scope, and separated collocation vs. boundary holdout data are non-negotiable, especially in safety-critical physical systems.

Technical Appendix — For ML/Scientific Computing Readers

Deep Dive: PINN Training & Complexity

★ This appendix targets readers with ML or scientific computing background. The main deck (Slides 1–10) is fully self-contained without it. Each section below corresponds to a "see Appendix" reference in the main slides.

A1 — PINN Training Stiffness & NTK-Based Loss Weighting (Wang et al., 2021)

The core training challenge is that L_data and L_physics produce gradients of vastly different magnitudes, causing one term to dominate and the other to stagnate — termed "gradient pathology" (Wang et al., 2021, SISC 43(5)). The fix is adaptive loss weighting based on Neural Tangent Kernel (NTK) analysis:

λ_opt(t) = max_θ |∇_θ L_data| / mean_θ |∇_θ L_physics| ← recomputed each training step

This ratio ensures both loss terms contribute equally to parameter updates. In practice, a running average smooths noisy per-step estimates. DeepXDE implements this automatically. Without NTK-based weighting, PINNs for the heat equation typically require 5–10× more epochs to reach equivalent accuracy, and often fail to converge for stiff PDEs entirely.

A2 — Causal Training for Time-Dependent PDEs (Wang et al., 2022)

Standard PINN training applies the physics loss uniformly across all time steps simultaneously. This is causally incorrect for parabolic PDEs like the heat equation — it demands the network satisfy constraints at t=10s before learning the solution at t=1s. Causal weighting:

L_causal = Σ_i w_i · L_PDE(t_i), w_i = exp(−ε · Σ_{j<i} L_PDE(t_j))

The weight w_i is suppressed until earlier time steps have low residual. ε controls the sharpness of the temporal progression. This approach (Wang et al., arXiv:2203.07404) reduces required training iterations by 5–10× for transient problems like the 2D heat system, and is essential for physically correct convergence on problems where initial conditions strongly constrain the solution trajectory.

A3 — Auto-Differentiation Cost Derivation for PDE Residuals

To evaluate ∂²u/∂x², automatic differentiation must traverse the computation graph twice (forward-over-reverse mode, i.e. Hessian-vector product). For a network with L layers of width d and n_c collocation points:

Cost_PINN ≈ n_c · (5 × C_fwd) + N_d · (2 × C_fwd) vs Cost_NN ≈ N · (2 × C_fwd)

The 5× factor per collocation point = 1 forward + 2 backward (first derivative) + 2 backward (second derivative, for second-order PDEs). For n_c = 10,000 and N_d = 200: PINN cost ≈ 50,400 equivalent forward passes vs 400 for a standard NN — a ~126× cost multiplier at this collocation density. This is why PINN training wall-clock time dominates despite architecturally similar networks.

A4 — Flash Attention: IO Complexity Reduction (Dao et al., 2022)

Standard self-attention materialises the full n×n score matrix S = QKᵀ/√d in GPU high-bandwidth memory (HBM), requiring O(n²·d) HBM reads/writes. The bottleneck is memory bandwidth, not arithmetic throughput.

Standard: HBM I/O ∝ n²·d | Flash Attn: HBM I/O ∝ n·d·(M)^(1/2)

Flash Attention tiles computation so Q, K, V blocks fit entirely in SRAM (L1 cache, ~20 MB on A100). The full attention matrix is never written to HBM. Output is bit-for-bit identical to standard attention — this is an IO-complexity result, not an approximation. For n=8k tokens: memory reduced from ~8 GB to ~300 MB at FP16. This is the primary engineering reason long-context models (>32k tokens) became practically feasible.

A5 — Collocation Scaling: How Many Points Are Needed? (Mishra & Molinaro, 2023)

The generalisation error of a PINN depends on both network capacity and collocation density. Mishra & Molinaro (2023, IMA J. Numer. Anal. 43(1)) establish the following practical guideline for second-order PDEs in d spatial dimensions:

n_c = O(ε^{-2(1+d/α)}) where α = Sobolev regularity of the PDE solution, ε = target error

For the 2D heat equation (d=2, α≈2, target ε=10⁻³): n_c ≈ O(10⁶) in the worst case. In practice, adaptive collocation sampling (concentrating points near sharp gradients and boundaries) reduces this by 10–100×, making 10k–50k points sufficient for engineering-grade accuracy. Always use adaptive sampling via a residual-based point resampler — DeepXDE's PDEPointResampler callback implements this with minimal code change.
