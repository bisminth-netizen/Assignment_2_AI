Choosing the wrong architecture for a physical system isn't just an academic mistake — it costs real compute time, real money, and produces models that hallucinate physics they were never taught.

Plain language — what this formula means A PINN trains on two signals simultaneously. L_data: how well predictions match actual sensor measurements. L_physics: how much the predictions violate the known governing equation (e.g. Newton's 2nd law, heat conduction). The parameter λ controls the balance between them. Unlike a standard neural network, the PINN is penalised every time a prediction contradicts the physics — making it impossible to "forget" the underlying laws of the system.
Classical NN / Transformer

Data-Driven Approach

Learns patterns purely from data. Powerful generalisation but blind to physical laws. Requires large labeled datasets. Scales with data volume and compute.

Physics-Informed NN (PINN)

Physics + Data Hybrid

Embeds governing equations (PDEs/ODEs) via Lphysics in the loss function. Extrapolates physically. Works with sparse sensor data. Training is harder due to competing, stiff loss gradients (see Appendix A1).

04 — Scaling Intuition

When does more = better?

Scaling laws describe how training loss decreases as data, parameters, or compute increase. PINNs and NNs follow different scaling curves — they cross at a regime boundary — and PINNs plateau earlier in the high-data regime due to training stiffness from competing loss gradients.

Low-Data Regime — PINN Wins
Physics acts as a massive implicit regulariser — the model cannot overfit to noise because the PDE loss anchors every prediction. This is the primary engineering use case for PINNs: expensive physical experiments yield only sparse sensor readings, but the governing equations fill in the spatial gaps.
High-Data Regime — NN Wins
Classical NNs and Transformers scale better with data. PINNs plateau earlier — the physics loss creates gradient conflict (Wang et al., 2021, SISC 43(5)). This is the training stiffness problem: two competing objectives prevent the model from squeezing out the final error that a pure data-driven model achieves freely.
Chinchilla Scaling Law Analogy (Hoffmann et al., 2022, arXiv:2203.15556)

For LLMs: optimal model size ∝ dataset size — scale both proportionally. For PINNs: optimal collocation point count ∝ PDE solution complexity. The principle is identical: do not blindly scale model parameters without proportionally scaling the training signal. Increasing network width without increasing n_c leads to physics overfitting; increasing n_c without wider networks underfits the boundary conditions.

Empirical basis for the scaling chart above The crossover point and plateau behaviour shown are conceptually derived from: (1) Wang et al. (2021, SISC) who show empirically that PINN loss stagnates due to gradient imbalance once data density exceeds a threshold; (2) Mishra & Molinaro (2023, IMA J. Numer. Anal.) who prove generalisation bounds showing PINN error decays slower than NN error as n → ∞ when physics residuals compete with data fit. The exact crossover is problem-specific — benchmark on your own system before drawing conclusions.

🚨 Data Leakage — What It Is & Why It's Catastrophic
  • Definition: When information from the test/validation set contaminates the training process. The model appears better than it is — it is effectively "cheating" on the exam.
  • Case 1 — Normalisation: Computing μ/σ across the entire dataset before splitting. Test distribution is baked into preprocessing. Fix: fit scalers only on training fold, then transform test with the same scaler.
  • Case 2 — Temporal split: In time-series physical systems (e.g., heat flow), a random train/test split leaks future states into training. Fix: always split by time — first 80% of timesteps for training, last 20% for testing.
  • Case 3 — PINN boundary conditions: Using the same boundary condition measurements in both the training physics loss and the validation metric. Fix: maintain a held-out boundary test set used only for evaluation.
  • Consequence: Validation loss looks excellent. Real-world performance collapses. Particularly dangerous in safety-critical systems (aerospace, nuclear, medical devices) where field failures are costly or catastrophic.
✓ Best Practice

Always fit preprocessing (scalers, encoders) only on training data. Use temporal or physics-domain-aware splits. Version your data with DVC or equivalent — data provenance is as important as code versioning.

PINN-Specific Rule

Strictly separate collocation points (interior domain, used in L_physics) from boundary condition sensor measurements (used in L_data). These must never overlap with the validation set. A dedicated boundary holdout set is mandatory for honest evaluation.

💡 Memory Bandwidth is the Real Bottleneck

In both architectures, the practical constraint is often memory bandwidth, not raw FLOPs. Matrix multiplications are memory-bound at small batch sizes — the GPU spends more time waiting for data from DRAM than performing arithmetic. PINN's collocation loop creates many small sequential operations, leading to poor GPU occupancy. Always profile with PyTorch Profiler or Nsight Compute before optimising.

Why PINN training steps are 3–5× more expensive A standard neural network gradient step = one forward pass + one backward pass. A PINN must additionally compute second-order derivatives of the output (e.g. ∂²u/∂x²) to evaluate the PDE residual. This requires running backpropagation through the backpropagation graph itself — second-order auto-differentiation. The full cost derivation, including the exact FLOPs multiplier formula, is in Appendix A3.
07 — Case Study: Physical System
🌡️
Chosen Physical System

2D Heat Conduction Problem

A steel plate (α = 1.17×10⁻⁵ m²/s) with a fixed heat source at one end and cooling boundary conditions. Classic PDE-governed physical system used in aerospace, electronics thermal management, and manufacturing process control.

∂u/∂t = α (∂²u/∂x² + ∂²u/∂y²) ← 2D heat equation (parabolic PDE)

Boundary conditions: u(0,y,t) = T_hot = 400 K, u(L,y,t) = T_cold = 300 K
Initial condition: u(x,y,0) = T_ambient = 293 K

PINN loss: L = ‖∂û/∂t − α∇²û‖² + λ‖ûBC − uBC,data‖²
Plain language — the heat equation for non-specialists The equation says: how fast temperature changes at any point = thermal diffusivity of steel (α) × how curved the temperature profile is around that point. The PINN learns to predict temperature û(x,y,t) at any location and time, penalised at every step for predictions that violate this relationship — not just at 20–50 sensors, but at 10,000+ interior domain points simultaneously.

08b — Critical Discussion

When PINNs Fail — And What To Do About It

PINNs are a powerful tool, but they are not universally superior. Understanding where they break down is as important as understanding where they shine. A responsible engineering recommendation must address both sides.

⚡ High-Frequency / Multi-Scale Solutions
Neural networks have a known "spectral bias" — they learn low-frequency components first and struggle with rapidly oscillating solutions. For turbulent flows or high-frequency vibration PDEs, standard PINNs converge to a smooth but incorrect low-frequency approximation. Fix: Fourier feature embeddings or multi-scale PINN architectures.
Ref: Rahaman et al. (2019, ICML) — spectral bias in deep networks
⚡ Stiff PDEs (sharp gradients)
When the PDE solution has sharp boundary layers (e.g., thin thermal boundary layers, shock fronts in compressible flow), the loss landscape becomes extremely ill-conditioned. Training can completely fail without adaptive collocation sampling near the sharp features. Fix: Residual-based adaptive refinement (RAR); concentrate collocation points near high-residual zones.
Ref: Lu et al. (2021, SIAM Rev.) — DeepXDE adaptive sampling
⚡ Long Time-Horizon Problems
Standard PINNs applied to long temporal windows (e.g., simulate a process for 1000 seconds) often suffer from catastrophic error accumulation — the physics residual explodes as time increases. The model learns to satisfy the PDE at early times but drifts arbitrarily at late times. Fix: Causal training (Wang et al., 2022) or domain decomposition in time ("XPINNs").
Ref: Wang et al. (2022, arXiv:2203.07404) — causal training
⚡ Inverse Problems with Noisy Data
While PINNs are often cited as ideal for inverse problems (inferring material parameters from sensor data), noise in boundary/sensor data is amplified by the second-order derivatives in the physics loss. A small error in measured u propagates to a large error in ∂²u/∂x². Fix: Gaussian process-based uncertainty quantification for noisy BCs; or Bayesian PINN variants.
Ref: Yang et al. (2021, JCP) — Bayesian PINNs for noisy inverse problems
⚡ High-Dimensional PDEs (d > 4)
The number of collocation points required to adequately cover the domain grows exponentially with dimension — the "curse of dimensionality." For d = 6+ spatial dimensions (e.g., kinetic equations, option pricing), standard PINNs become computationally intractable. Fix: Deep BSDE methods, or quasi-Monte Carlo collocation sampling for high-d.
Ref: Mishra & Molinaro (2023, IMA J. Numer. Anal. 43(1)) — error bounds scale with dimension
⚡ Gradient Pathology — L_physics Dominates
If λ is too large, the physics gradient completely overwhelms the data gradient — the model satisfies the PDE beautifully but ignores sensor measurements entirely. If λ is too small, the reverse occurs. Manual λ tuning is brittle and domain-specific. Fix: NTK-based adaptive λ scheduling recomputed each step (Wang et al., 2021, SISC 43(5)).
Ref: Wang et al. (2021, SISC 43(5)) — NTK-based loss weighting
✅ When PINNs Work Best (Summary)
✓ Sparse sensor data (<1k points) on well-understood PDE
✓ Smooth solutions without sharp gradients or multi-scale oscillations
✓ Short-to-medium time horizons (t < 100 characteristic times)
✓ Safety-critical systems where physics compliance is non-negotiable
✓ Low-dimensional spatial domains (d ≤ 3) with well-posed BCs
✓ Inverse problems with low noise and known governing equations
09 — Summary & Takeaways

The 3 Things engineering management needs to know

01

Architecture is a Data Decision

Don't pick NN vs PINN based on hype or familiarity. Ask: how much labeled data is available, and do we need physics compliance? Classical NNs need data volume. PINNs need governing equations. The right answer changes entirely with the data regime.

02

Matrix Ops = Cloud Bill

O(n²) attention and O(n_c × L × d²) PINN auto-diff translate directly to compute cost. PINN steps are 3–5× more expensive due to second-order differentiation (Raissi et al., 2019). Always profile before optimising — memory bandwidth is usually the real constraint.

03

Data Leakage Kills Silently

Models with leakage look excellent in validation and fail in production. Temporal splits, correct normalisation scope, and separated collocation vs. boundary holdout data are non-negotiable in safety-critical physical systems.

The Honest Bottom Line

For physical systems with sparse data and hard physics constraints, PINNs deliver demonstrably better extrapolation and physics compliance — our mock benchmark shows 1.8% vs. 7.2% L2 error in the sparse-sensor regime and 15× lower physics residual than a classical NN. But PINNs are not a free lunch: they cost 3–5× more to train, fail on high-frequency and long-horizon problems, and require careful λ scheduling to converge. The engineering decision is not "which is better?" — it is "which is better for this data regime, this compute budget, and this physics?" Start with PINN when data is scarce and physics is known. Scale to NN surrogates when throughput matters. Profile everything before committing to infrastructure.

Sparse data → PINN Large data → NN Real-time → Distil to surrogate NN Always profile first

Technical Appendix — For ML/Scientific Computing Readers

Deep Dive: PINN Training & Complexity

★ This appendix targets readers with ML or scientific computing background. The main deck (Slides 1–10) is fully self-contained without it. Each section below corresponds to a "see Appendix" reference in the main slides.

A1 — PINN Training Stiffness & NTK-Based Loss Weighting (Wang et al., 2021, SISC 43(5))

The core training challenge is that L_data and L_physics produce gradients of vastly different magnitudes, causing one term to dominate and the other to stagnate — termed "gradient pathology." The fix is adaptive loss weighting based on Neural Tangent Kernel (NTK) analysis:

λ_opt(t) = max_θ |∇_θ L_data| / mean_θ |∇_θ L_physics| ← recomputed each training step

This ratio ensures both loss terms contribute equally to parameter updates. DeepXDE implements this automatically. Without NTK-based weighting, PINNs for the heat equation typically require 5–10× more epochs to reach equivalent accuracy, and often fail to converge for stiff PDEs entirely.

A2 — Causal Training for Time-Dependent PDEs (Wang et al., 2022, arXiv:2203.07404)

Standard PINN training applies the physics loss uniformly across all time steps simultaneously. This is causally incorrect for parabolic PDEs like the heat equation — it demands the network satisfy constraints at t=10s before learning the solution at t=1s. Causal weighting:

L_causal = Σ_i w_i · L_PDE(t_i), w_i = exp(−ε · Σ_{j<i} L_PDE(t_j))

The weight w_i is suppressed until earlier time steps have low residual. ε controls the sharpness of the temporal progression. This approach reduces required training iterations by 5–10× for transient problems like the 2D heat system.

A3 — Auto-Differentiation Cost Derivation for PDE Residuals (Raissi et al., 2019, §3)

To evaluate ∂²u/∂x², automatic differentiation must traverse the computation graph twice (forward-over-reverse mode). For a network with L layers of width d and n_c collocation points:

Cost_PINN ≈ n_c · (5 × C_fwd) + N_d · (2 × C_fwd) vs Cost_NN ≈ N · (2 × C_fwd)

The 5× factor per collocation point = 1 forward + 2 backward (first derivative) + 2 backward (second derivative). For n_c = 10,000 and N_d = 200: PINN cost ≈ 50,400 equivalent forward passes vs 400 for a standard NN — a ~126× cost multiplier at this collocation density.

A4 — Flash Attention: IO Complexity Reduction (Dao et al., NeurIPS 2022)

Standard self-attention materialises the full n×n score matrix S = QKᵀ/√d in GPU HBM, requiring O(n²·d) HBM reads/writes. The bottleneck is memory bandwidth, not arithmetic throughput.

Standard: HBM I/O ∝ n²·d | Flash Attn: HBM I/O ∝ n·d·(M)^(1/2)

Flash Attention tiles computation so Q, K, V blocks fit entirely in SRAM. The full attention matrix is never written to HBM. Output is bit-for-bit identical to standard attention — this is an IO-complexity result, not an approximation. For n=8k tokens: memory reduced from ~8 GB to ~300 MB at FP16.

A5 — Collocation Scaling (Mishra & Molinaro, 2023, IMA J. Numer. Anal. 43(1))

The generalisation error of a PINN depends on both network capacity and collocation density. Practical guideline for second-order PDEs in d spatial dimensions:

n_c = O(ε^{-2(1+d/α)}) where α = Sobolev regularity of the PDE solution, ε = target error

For the 2D heat equation (d=2, α≈2, target ε=10⁻³): n_c ≈ O(10⁶) worst case. In practice, adaptive collocation via residual-based point resampling (DeepXDE's PDEPointResampler) reduces this to 10k–50k points for engineering-grade accuracy.
